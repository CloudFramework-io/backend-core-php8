# CLOUD Documentum

CLOUD Documentum is CloudFramework's documentation management system for APIs, Libraries, Processes, WebApps, Courses, and organizational knowledge.

---

## Quick Reference

### Mandatory Rules

1. **Always backup before modifying**: Run `backup-from-remote` before any modification
2. **No KeyId for new records**: New ENDPOINTs, Modules, Contents, SubProcesses, **Checks** auto-generate KeyId on sync
3. **Platform from config**: Platform is read from `core.erp.platform_id`, not URL path
4. **Checks require CFOField and Route**: When creating Checks, `CFOField` must be `"JSON"` and `Route` must match the parent's JSON route

### ⚠️ CRITICAL: KeyId Behavior for New Child Records

When creating **NEW** child records (Modules, ENDPOINTs, Contents, SubProcesses, Checks), you **MUST NOT** include the `KeyId` field. The KeyId is **auto-generated by the remote server** when the record is inserted.

**Applies to:**
- `CloudFrameWorkDevDocumentationForWebAppsModules` (WebApp Modules)
- `CloudFrameWorkDevDocumentationForAPIEndPoints` (API ENDPOINTs)
- `CloudFrameWorkDevDocumentationForLibrariesModules` (Library Modules)
- `CloudFrameWorkDevDocumentationForSubProcesses` (SubProcesses)
- `CloudFrameWorkDevDocumentationForProcessTests` (Checks)
- `CloudFrameWorkAcademyContents` (Course Contents)

**Workflow for new records:**
1. Create the record in local backup **WITHOUT KeyId**
2. Run `update-from-backup` → Script uses POST for records without KeyId
3. Run `backup-from-remote` → Gets the auto-generated KeyIds from remote
4. Local backup is now updated with the real KeyIds

**Example - New WebApp Module (correct):**
```json
{
    "WebApp": "/cloud-solutions/hrms",
    "EndPoint": "/new-feature",
    "Title": "New Feature Module",
    "Status": "2.IN DEVELOPMENT",
    "TeamOwner": "dev@example.com"
}
```

**Example - WRONG (do NOT include KeyId):**
```json
{
    "KeyId": "12345",  ← WRONG! Never add KeyId for new records
    "WebApp": "/cloud-solutions/hrms",
    ...
}
```

### Script Commands Pattern

All CloudIA scripts follow this pattern:
```bash
composer script -- _cloudia/{type}/{action}?{params}
```

| Type | Actions | Parameters |
|------|---------|------------|
| `apis` | `list-remote`, `list-local`, `backup-from-remote`, `insert-from-backup`, `update-from-backup` | `id=/api/path` |
| `libraries` | Same as above | `id=/library/path` |
| `processes` | Same as above | `id=PROCESS-ID` |
| `webapps` | Same as above | `id=/webapp/path` |
| `checks` | Same as above | `entity=CFOEntity&id=CFOId` |
| `courses` | Same as above | `id=COURSE-KEYID` |
| `cfos` | Same as above | `id=CFOKeyName` |
| `webpages` | Same as above | `id=PAGE-ID` |
| `resources` | Same as above | `id=RESOURCE-KEY` |
| `menu` | Same as above | `id=MODULE-KEY` |

## Check Documentation

Checks are tests, objectives, and specifications linked to other documentation elements.

### Mandatory Fields for Checks

When creating Checks, these fields are **mandatory**:

| Field | Description | Required Value |
|-------|-------------|----------------|
| `CFOEntity` | CFO KeyName of parent entity | e.g., `CloudFrameWorkDevDocumentationForWebApps` |
| `CFOId` | KeyName/KeyId of parent record | e.g., `/cloud-documentum/webapps-modules` or `5155462638469120` |
| `CFOField` | Field containing the JSON with routes | Always `"JSON"` for documentation CHECKs |
| `Route` | Unique path for JSON linking | e.g., `/header-gradient-card` |

> **CRITICAL**:
> - **Never include `KeyId`** when creating new Checks - it auto-generates on sync
> - **`CFOField` must be `"JSON"`** to properly link with the parent's JSON field
> - **`Route` must be unique** within the same CFOEntity/CFOId combination

### Linking System

Checks link to parent entities via:

| Field | Description | Example |
|-------|-------------|---------|
| `CFOEntity` | CFO KeyName of parent | `CloudFrameWorkDevDocumentationForProcesses` |
| `CFOId` | KeyName/KeyId of record | `cloud-hrms` or `5077124951572480` |
| `CFOField` | Parent field with routes | `JSON` |
| `Route` | Reference for JSON linking | `/setup-guide` |

### Route Mechanism

```
Process.JSON: {"Setup Guide": {"route": "/setup-guide"}}
        │
        └──> Check.Route = "/setup-guide"
             Check.CFOEntity = "CloudFrameWorkDevDocumentationForProcesses"
             Check.CFOId = "cloud-hrms"
```

### WebApp Documentation Workflow

Complete workflow for documenting a WebApp from scratch with Checks verification system.

#### Step 1: Create WebApp Backup File

Create the backup file in `buckets/backups/WebApps/{platform}/`:

```json
{
    "CloudFrameWorkDevDocumentationForWebApps": {
        "KeyName": "/verticals/hrms/employee-queries",
        "Title": "CloudIA Employee Queries",
        "Description": "<p>AI assistant for employee queries</p>",
        "Status": "2.IN DEVELOPMENT",
        "TeamOwner": "dev@example.com",
        "CloudFrameworkUser": "creator@example.com",
        "JSON": {
            "Scope": {
                "General Scope": {"route": "/scope-general"},
                "Query Types": {"route": "/scope-queries"},
                "Architecture": {"route": "/scope-architecture"}
            },
            "Implementation": {
                "Frontend WebApp": {"route": "/impl-frontend"},
                "Backend API": {"route": "/impl-backend"},
                "AI Integration": {"route": "/impl-ai"}
            }
        }
    },
    "CloudFrameWorkDevDocumentationForWebAppsModules": []
}
```

#### Step 2: Add Technical Modules

Add modules to the `CloudFrameWorkDevDocumentationForWebAppsModules` array:

```json
{
    "WebApp": "/verticals/hrms/employee-queries",
    "EndPoint": "/frontend",
    "Title": "Frontend WebApp Module",
    "Folder": "Technical Modules",
    "Status": "2.IN DEVELOPMENT",
    "TeamOwner": "dev@example.com",
    "CloudFrameworkUser": "creator@example.com",
    "DateInsertion": "2026-01-12 10:00:00",
    "JSON": {
        "Components": {
            "Chat Interface": {"route": "/component-chat"},
            "Query History": {"route": "/component-history"}
        },
        "Verifications": {
            "Responsive Design": {"route": "/check/responsive"},
            "Accessibility": {"route": "/check/accessibility"}
        }
    }
}
```

> **Note**: Do NOT include `KeyId` - it auto-generates on sync.

#### Step 3: Sync WebApp to Remote

```bash
# Insert new WebApp
composer script -- "_cloudia/webapps/insert-from-backup?id=/verticals/hrms/employee-queries"

# Or update existing
composer script -- "_cloudia/webapps/update-from-backup?id=/verticals/hrms/employee-queries"
```

#### Step 4: Backup to Get KeyIds

After sync, backup to get auto-generated KeyIds for modules:

```bash
composer script -- "_cloudia/webapps/backup-from-remote?id=/verticals/hrms/employee-queries"
```

#### Step 5: Create Checks for WebApp

Create checks file in `buckets/backups/Checks/{platform}/`:

**Filename**: `CloudFrameWorkDevDocumentationForWebApps___verticals_hrms_employee-queries.json`

```json
{
    "CFOEntity": "CloudFrameWorkDevDocumentationForWebApps",
    "CFOId": "/verticals/hrms/employee-queries",
    "CloudFrameWorkDevDocumentationForProcessTests": [
        {
            "CFOEntity": "CloudFrameWorkDevDocumentationForWebApps",
            "CFOField": "JSON",
            "CFOId": "/verticals/hrms/employee-queries",
            "Route": "/scope-general",
            "Title": "General Scope Definition",
            "Description": "<p>Define the general scope...</p>",
            "Status": "pending",
            "Owner": "dev@example.com"
        },
        {
            "CFOEntity": "CloudFrameWorkDevDocumentationForWebApps",
            "CFOField": "JSON",
            "CFOId": "/verticals/hrms/employee-queries",
            "Route": "/impl-frontend",
            "Title": "Frontend Implementation",
            "Description": "<p>Frontend requirements...</p>",
            "Status": "pending",
            "Owner": "dev@example.com"
        }
    ]
}
```

> **Important**: Each check must include `CFOEntity`, `CFOField: "JSON"`, `CFOId`, and `Route`. Do NOT include `KeyId`.

#### Step 6: Create Checks for Modules

Create checks for each module using the KeyId from Step 4:

**Filename**: `CloudFrameWorkDevDocumentationForWebAppsModules__5077124951572480.json`

```json
{
    "CFOEntity": "CloudFrameWorkDevDocumentationForWebAppsModules",
    "CFOId": "5077124951572480",
    "CloudFrameWorkDevDocumentationForProcessTests": [
        {
            "CFOEntity": "CloudFrameWorkDevDocumentationForWebAppsModules",
            "CFOField": "JSON",
            "CFOId": "5077124951572480",
            "Route": "/check/responsive",
            "Title": "Responsive Design Verification",
            "Description": "<p>Verify responsive design works correctly</p>",
            "Status": "pending",
            "Owner": "dev@example.com"
        }
    ]
}
```

> **Important**: Each check in modules must also include `CFOEntity`, `CFOField: "JSON"`, `CFOId`, and `Route`. Do NOT include `KeyId`.

#### Step 7: Sync All Checks

```bash
# Sync WebApp checks
composer script -- "_cloudia/checks/update-from-backup?entity=CloudFrameWorkDevDocumentationForWebApps&id=/verticals/hrms/employee-queries"

# Sync Module checks (use KeyId)
composer script -- "_cloudia/checks/update-from-backup?entity=CloudFrameWorkDevDocumentationForWebAppsModules&id=5077124951572480"
```

### Checks in WebApps - Best Practices

1. **Categorize logically**: Use categories like `"Scope"`, `"Implementation"`, `"Verifications"`

2. **Use route prefixes**: Use `/check/` for verification checks vs `/impl-` for documentation
   ```json
   {"Verification": {"route": "/check/responsive"}}
   {"Documentation": {"route": "/impl-frontend"}}
   ```

3. **Link at appropriate level**:
   - **WebApp level**: General acceptance criteria
   - **Module level**: Feature-specific checks

4. **Check states**: `pending` → `in-progress` → `blocked` / `in-qa` → `ok`

5. **Module JSON structure**:
   ```json
   {
       "Documentation": {"Feature Docs": {"route": "/docs"}},
       "Verifications": {"Test 1": {"route": "/check/test-1"}}
   }
   ```
